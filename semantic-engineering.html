The discipline behind dependable AI

Semantic Engineering

The discipline that grounds AI agents in a governed model of how your business works, so enterprise AI is dependable in production, auditable on demand, and yours to keep.

Refined since 2017Runs inside Breeze.AI and ASIMOVFull methodology at semantic-engineering.ai
What it is

AI that knows how your business works

Most AI struggles at enterprise scale because it has no structured way to know how your business works, so it fills the gaps by guessing. Semantic Engineering closes that gap. We encode the knowledge your business runs on, its rules, structures, decisions, and relationships, into a governed, queryable model, and put agents to work against it, with every action validated and every change traceable.

The method is a loop. Your knowledge is encoded into a governed model. Agents act against that model instead of guesswork. Every change passes a validation gate that leaves evidence, and named human custodians keep the model honest as the business moves. We have been refining this discipline since 2017, and it sits inside every offering on this site; the engines on Our Core are how it reaches production.

The problem it removes

The manual translation tax

Enterprise software work is hard because the knowledge that holds a system together lives in heads, in decaying documents, and in code that is opaque without interpretation. AI coding assistants handle small, contained tasks well and break against this complexity, because they have no structured way to query the system's real state.

Bigger context windows do not fix this. The agent needs structured context it can read against, not more raw text to wade through. So every task pays a recurring cost: someone converts tacit knowledge into a form the AI can act on, by hand, again and again. We call that cost the manual translation tax, and removing it is the point of the discipline. Greenfield, brownfield, and legacy modernization all hit the same wall; the complexity differs, the structural gap does not.

Read the business case in The manual translation tax, one of the methodology adaptations in our Insights library.

Where the knowledge lives

Four custodians carry the picture. No one carries all of it.

In a typical enterprise application, four roles hold the tacit knowledge, and no single person or document holds the whole. Semantic Engineering names each role the accountable custodian of one layer of the model, so decay is treated as an ownership failure, not a tooling problem.

The Product Owner holds the why

Functional layer

Personas, outcomes, scenarios, and the proposals that were rejected along the way. As custodian, the product owner curates the functional layer of the model.

The Architect holds the where

Architecture layer

Service boundaries, source-of-truth databases, integration contracts. The architect curates the architecture layer.

The UX Designer holds the look and behavior

Design layer

Components, interaction patterns, state handling. The UX designer curates the design layer.

The Engineering Team holds what the code does

Code layer

Live functions, retry policies, active feature flags, dead utilities. The engineering team curates the code layer, which updates as a side effect of every merge.

The shared source of truth

Ontologies: one model, one valid interpretation

Those four layers of knowledge become a four-layer ontology: a machine-readable, interconnected model that admits only one valid interpretation and maintains traceable connections across every layer. Agents traverse it to pull only the context a change requires.

Personas · Outcomes · Scenarios · Custodian: Product Owner
Design

Functions · Feature flags · Retry policies · Custodian: Engineering Team

Agents generate only against what the model asserts. A function an agent references must exist in the code layer; a boundary it crosses must be declared in the architecture layer. Hallucinations get caught at a structural gate before merge, not in production.

The shape of the model follows the work. Live software, greenfield or brownfield, uses this four-layer ontology, curated continuously. Legacy modernization has a fixed target, so it uses a source-state model decomposed from the legacy code, a target-state model defined by the same four custodians, and a specification format that bridges the two. The principles, the custodianship, and the gates are identical in every shape. For the layer-by-layer walkthrough, read The four-layer ontology.

The discipline

Four principles, enforced on every engagement

A structured model

We encode the knowledge a business runs on as a queryable model, so agents work from real context instead of inference.

Agents bound by the model

Agents generate and act only against what the model asserts. They cannot invent capabilities or facts that are not there.

Named custodianship

Every part of the model has a named human custodian who is accountable for keeping it honest.

Validation gates with evidence

Every change is checked at a gate that emits machine-verifiable evidence, so quality is proven, not assumed.

What it gives you

Three things follow from working this way

Dependability

Agents act with the context a task needs, so they hold up in production at enterprise scale, where ungrounded AI breaks down.

Auditability

Every action passes a gate and leaves evidence, so you can show exactly what happened, why, and against which version of the model.

Ownership

The model is your asset, captured from your business and maintained by your own people. The value compounds and stays with you as the model gets richer over time.
