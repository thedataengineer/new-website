A Four-Layer Ontology for AI-Native Software Delivery

AI agents cannot make safe changes from a pile of documents. They need a queryable model of the system as it runs. This is the structure we build it in.

Ask an AI agent to make a safe, targeted change inside a 1.6 million line application and it needs to know which functions and modules carry the relevant behavior. Your documents cannot tell it: documentation describes the system as designed, and specs describe the system as the team wishes it were. The Semantic Knowledge Graph at the center of our Semantic Engineering methodology describes the system as it runs, captured as four cross-linked ontology layers that stay current on every merge.

Why prompt stuffing fails first

Without a structured substrate, the only way to brief an agent is to feed it enough raw text to cover every case it might meet: the codebase plus the spec library. The richer the change, the larger the prompt. The larger the prompt, the higher the per-invocation cost, and the more likely the agent misses something buried in the middle, which triggers another round with more context. Token spend grows super-linearly with change complexity.

A queryable graph inverts the economics. Each agent invocation pulls only the slice its task needs: the affected personas and outcomes, the architecture nodes the change crosses, the code modules involved. The rest stays in storage. Cost per change tracks the size of the change, not the size of everything around it.

Four layers, four named owners

Frederick Brooks called the shared theory of a system its design concept: the understanding everyone working on it carries in their head. The graph is that design concept made persistent and queryable instead of reconstructed in every conversation. It has four layers, each with a custodian accountable for keeping it honest.

  • Functional, owned by the product owner: personas, outcomes, scenarios, steps, and actions. When someone says users should be able to do X, this layer forces the question: which persona, which outcome, which scenario?
  • Design, owned by the UX designer and design system owner: user journeys, flows, and pages on one side; templates, organisms, molecules, and atoms on the other. It surfaces existing components before a team builds a duplicate and exposes drift between Figma and the running UI.
  • Architecture, owned by the architect and tech lead: an eight-layer blueprint spanning user experience, API gateway, services, agents, event queues, data stores, observability, and infrastructure. Boundary violations, such as a frontend calling a service straight past the gateway, surface as graph-shape violations during PR validation.
  • Code, owned by the engineering team: files, classes, functions, statements, and API endpoints, auto-generated from source by AST parsers on every merge to master, across ten supported languages including TypeScript, Java, Python, and C#.

One rule binds all four. Every node carries a citation to the source document, Figma frame, Jira ticket, or code file that justifies its existence; a node without a citation does not enter the graph. That makes the graph audit-grade, and it lets agents diff the graph against its sources: when a ticket changes, agents know which nodes to re-validate.

What a real traversal returns

A product owner files a user story: let users choose daily, weekly, or no email alerts for each saved search. Before anyone writes code, the Impact Analysis Agent traverses all four layers and returns the full blast radius. Two outcomes modified and two scenarios added on the functional side. Three components and one new email template in design. Fourteen architecture nodes touched, including a new task family per region. Fifteen specific code changes across five repositories, plus the database column that already exists, so no schema change is required. The senior developer who used to assemble that picture by hand now reviews it instead.

On a 1.6 million line application that query completes in about eight minutes. That number drives a structural decision: one graph per product, not a monolith across the portfolio. Monolithic graphs slow until queries stop being useful; per-product graphs stay fast, and specialized agents consult multiple graphs in sequence when a question crosses products.

Nobody writes this by hand

Brownfield adoption starts with extraction, not authoring. AST parsers generate the code layer. The architecture layer is inferred from code structure and cross-validated against whatever documentation exists. The functional layer is inferred from the enriched code graph, starting at the UI and following the code paths. The design layer comes from component code and wireframes, enriched by browser-automation agents that exercise the application end to end. For an application above two million lines of code, extraction typically completes in two to three weeks.

Extraction also doubles as a rationalization audit the client did not have to budget for. Four patterns surface mechanically:

  • Duplicate capabilities. The same outcome built two or three times in different code paths, with no one aware the duplicates exist.
  • Split functionality. One outcome scattered across repositories with no shared abstraction and undocumented coordination between them.
  • Dead capabilities. Features still carried in the codebase that no live code path reaches.
  • Misclassified architecture. A service named billing that owns most of the user-profile logic.

One client had organized its backends around functional domains, one team per domain, behind a single unified front end. After four to five years, the same outcomes had been implemented two or three times across those products. Extraction made the duplication visible and produced a prioritized cleanup backlog that flowed into normal sprint planning.

What keeps it from rotting

A graph nobody measures degrades the way wiki pages degrade. So the graph is governed like a production system. Fourteen verification checks run in CI; the highest-priority ones run on every merge and block it on failure. Twenty-nine metrics across six categories track growth, structure, connectivity, and architectural drift on release and quarterly cadences. A freshness rule enforces currency: any layer stale beyond thirty days auto-triggers a refresh sprint.

Scope discipline matters as much as measurement. Teams that try to put everything into the graph drown in maintenance within a few sprints. The inclusion test is blast radius: an element enters the graph only when changing it would cascade across teams or layers. Personas, service boundaries, and integration contracts go in. Button copy, local styling overrides, and private helper methods stay out. The default is exclusion; the aperture widens deliberately over the first dozen sprints.

How it becomes operational

Our Breeze.AI platform maintains the product-partitioned graphs and runs the extraction, the metrics and verification framework, and the three impact analysis variants. Managed support tiers, Light through Deep, set how much of that cadence Accion runs alongside your own custodians.
