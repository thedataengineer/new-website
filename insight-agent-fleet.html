The Agent Fleet: How Continuous Software Development Runs

A small fleet of specialized agents, each with a named human owner, turns the enterprise knowledge graph into a running delivery system.

Insight · Semantic EngineeringAgentic SDLC

Semantic Engineering runs continuous software development through a small fleet of specialized agents operating on a four-layer knowledge graph of your software: what it does, how it is designed, how the code is structured, and how the architecture fits together. Six to eight agents cover the whole operating model. Every one of them has a named human owner, and none of them writes to a governed record without human approval.

Small by design

Adding agents feels like progress. In practice, every additional agent adds operational surface area, so the fleet stays deliberately small and we resist agent proliferation. Seven agent classes cover the model:

  • Impact Analysis Agent. Reads a spec, traverses the knowledge graph, and produces a structured impact report before code is written and again after it merges.
  • PR Validation Agent. Gates every merge against all four ontology layers and refuses changes that violate the structure.
  • BDD Generation Agent. Generates test scenarios directly from the Functional Ontology.
  • KG Sync Agent. Updates the knowledge graph on every merge so it never goes stale.
  • Extraction Agents. Build the initial graph from an existing codebase, one agent per ontology layer.
  • Cross-Product Impact Extension. Runs impact analysis when a change spans multiple products.
  • Portfolio Rationalization Agent. Detects duplicated and dead capabilities across the portfolio on a quarterly cadence.

A two-level pattern coordinates them. One top-level orchestrator runs per engagement and holds cross-product policy; sub-orchestrators run per workstream and dispatch agents in the right sequence. Adding a workstream means adding a sub-orchestrator, not rewiring the top level, so coordination overhead stays manageable as the engagement grows.

Eight minutes versus five days

The Impact Analysis Agent is the clearest demonstration of what the graph buys you. A product manager writes a forty-line user story in an afternoon. Eight minutes later, the agent returns a report identifying which functional outcomes change, which UI components and code modules need modification, which database tables are affected, and which downstream services consume the change, with a cross-layer traceability matrix, single points of failure, a risk taxonomy, a QA test plan, and a suggested ticket-slicing plan. A senior engineer who knows the codebase well would need three to five working days to produce a comparable analysis, and would still miss several cross-cutting concerns.

On a real 1.6 million line Node.js, TypeScript and React application, a story about letting users pick daily or weekly email alerts produced a fifteen-section report with three findings that would have been expensive to reach manually. The database column for alert frequency already existed, which reclassified the work from schema migration to application change only. The existing daily worker would have sent weekly subscribers both a daily email and a weekly digest unless a filter was added; that defect would have shipped and surfaced as a customer incident. And three repositories had to deploy in a specific order, with out-of-band steps that normally live only in the heads of long-tenured engineers.

Bounding the objection every CTO raises

AI code generation is nondeterministic by default: two runs against the same prompt produce different code. The objection is real, and the fleet answers it by bounding the variance rather than claiming to remove it. With the impact report attached to the prompt, the structural decisions (which service, which components, which tables) are fixed in advance. What still varies is implementation detail: variable names, the precise shape of a function. That variance is acceptable. Code review shifts from asking whether the agent touched the right files to asking whether the implementation detail is correct, which is the judgment call review was for in the first place.

Enforcement at the merge, upkeep on every commit

The PR Validation Agent brackets the other end of the cycle. Every merge is checked against all four ontology layers: does the change touch what the spec said it would, does it reuse components that already exist, does it respect service boundaries, does it break downstream consumers. When a change in one repository removes an API field another team's code depends on, the gate blocks the merge before integration, when coordination is cheap, instead of after, when it is a production incident. The gate checks structure, not style; linting and test coverage stay in the conventional CI pipeline. Overrides are allowed, logged, and reviewed quarterly by the Chief Architect. The friction is deliberate: a single prevented incident typically costs more than a year of gate blocks for that class of violation.

Two more agents keep the system honest. BDD scenario writing is the discipline every team abandons under deadline pressure; scenarios go stale by the third sprint and get dropped by the sixth. The BDD Generation Agent produces Gherkin scenarios directly from the Functional Ontology and regenerates them on every update, so there is nothing to maintain by hand. Teams operating this way typically reach 93.4 percent test coverage with zero manual authoring effort, and the remaining gap is intentional. The KG Sync Agent updates the graph on every merge, in seconds, even on that 1.6 million line application. Run sync quarterly instead, and within one quarter the graph drifts far enough that impact reports become unreliable and teams start working around them.

Trust is earned one level at a time

The most common failure in enterprise AI adoption is the leap of faith: an agent gets broad authority based on demo performance, and its failure modes appear in production. Progressive autonomy is the structural answer. Every agent starts at Level 1, where it can only suggest, and moves through five levels toward autonomous execution. Promotion requires evidence. Every recommendation's disposition is logged, and the accuracy rate must clear a defined threshold, typically 95 percent for routine task classes and higher for safety-critical ones. Each promotion is captured in a Promotion Agreement naming the approver, the evidence, and the rollback criteria. Demotion is automatic when failure rates cross the threshold, and a Sev-1 incident traced to an agent sends it back to Level 1. A demotion means the discipline is working as designed.

Named humans, all the way down

Ownership is the governing principle. The Impact Analysis Agent belongs to the workstream's forward-deployed engineer, the PR gate and BDD diffs to the Tech Lead, graph sync to the Ontology Maintainer, cross-product analysis to the Chief Architect. The owner reviews outputs, approves writes to systems of record, and decides when the agent needs retraining. Today, engineers still write the code, with AI assistance; the fleet supplies the structured context that makes that code reliable. The stated endpoint sits further out: stop writing code and write only agents, rolled out on selected workstreams where the operating model can be redesigned alongside the tooling. Accion Labs operationalizes the fleet through the Breeze.AI platform, with a named owner from the engagement team for each agent in production.
